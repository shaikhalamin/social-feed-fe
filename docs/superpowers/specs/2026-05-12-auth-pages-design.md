# Auth Pages (Phase A) — Design

**Date:** 2026-05-12
**Status:** Approved (awaiting user review of written spec)
**Phase of:** Three-phase rollout — A: Design system + Auth pages (this spec) → B: Authenticated app shell → C: Feed page

## Purpose

Bootstrap the design system layer (shadcn/ui primitives, theme tokens, `cn` helper) and ship the two auth pages — `/auth/login` and `/auth/signup` — with real API integration. Visual target is pixel-faithful match to `sample_screens/login.html` and `sample_screens/registration.html`, rebuilt with Tailwind + shadcn (no Bootstrap, no custom `_social_login_*` CSS).

Phase A replaces empty-shell route bodies and the bare centered-card `auth.tsx` layout. Auth infrastructure (token store, refresh hook, route guards, `safeRedirectPath`) is already in place and is **not modified** in this phase.

## Non-Goals

- Dark mode and theme toggle (deferred to Phase B — the toggle lives in the header).
- Authenticated app shell, header, sidebars (Phase B).
- Feed list and post composer (Phase C).
- Real Google OAuth (button is decorative; toast on click).
- Forgot-password flow (link is decorative; toast on click).
- "Remember me" persistence (checkbox is decorative; no localStorage change to refresh-token TTL).
- Multi-tab login propagation (already noted as a known limitation; not added here).
- Email confirmation / verification flow.
- Card, Avatar, Dialog, Dropdown, Tabs, Skeleton, etc. — only the primitives the two auth forms need.
- Unit / integration tests (manual smoke gate suffices for Phase A's surface area).

## Architectural Decisions

| Decision | Choice | Rationale |
|---|---|---|
| UI library | shadcn/ui (style `new-york`, base `slate`, CSS variables) | User-mandated. Generated source under our control; aligns with Tailwind v4. |
| Theme primary color | `#377DFF` from sample | Exact-match user requirement. |
| Font | Inter Variable (already loaded) | Keep diff small; swap to Poppins only if visual review flags. |
| Form library | TanStack Form | Already in deps; integrates with TanStack Router. |
| Validation | Kubb-generated zod schemas from `@/gen/api/zod` | Single source of truth with backend contract. |
| Form recipe | Custom port of shadcn `<Form>` to TanStack Form API | Shadcn's published recipe is react-hook-form; minimal port (~30 lines). |
| Post-auth navigation | `window.location.href = …` (hard reload) | Matches existing `logoutCurrentDevice`; eliminates race between Zustand write and router re-eval. |
| Error display | Field-level zod errors inline; 401 login inline + toast; 409 signup inline + toast; other 4xx/5xx toast-only | Distinguishes user-correctable (inline) from server-side (toast). |
| Disabled feature affordances | Render visually + `toast.info("…coming soon")` on click | Pixel-faithful design without dead clicks. |
| Decorative SVGs | Copy `sample_screens/assets/images/{shape1,2,3,dark_shape*,login,registration,registration1,google}.{svg,png}` and `logo*.svg` to `public/auth/` and `public/` | Pixel-faithful design. Dark variants copied but unused this phase. |
| Module placement | `src/features/auth/` for form logic + mutations; `src/components/auth/` for visual layout pieces; `src/components/ui/` for shadcn primitives | Feature owns behavior; components are presentational; UI is generated. |

## Components & Responsibilities

Each module has one clear job, narrow inputs, and can be reasoned about in isolation.

### Foundation

**Purpose:** shadcn CLI config.
**Settings:** `style: new-york`, `baseColor: slate`, `cssVariables: true`, `aliases: { components: "@/components", utils: "@/lib/utils", ui: "@/components/ui" }`, Tailwind v4 mode (no `tailwind.config.js`).

#### `src/lib/utils.ts`
**Purpose:** `cn(...inputs)` → `twMerge(clsx(inputs))`. Generated by shadcn init.
**Exports:** `cn`.

#### `src/index.css` (modified)
**Purpose:** Append shadcn `@theme` token block. Variables tuned to sample palette:
- `--primary: #377DFF`, `--primary-foreground: #FFFFFF`
- `--background: #F5F7FB`, `--foreground: #1A1F36`
- `--card: #FFFFFF`, `--card-foreground: #1A1F36`
- `--muted-foreground: #666666` (matches sample's icon `fill="#666"`)
- `--border: rgba(0,0,0,0.08)`, `--input: rgba(0,0,0,0.12)`
- `--ring: #377DFF`
- `--radius: 0.5rem`

Existing `--font-sans` and base body styles are preserved.

### UI Primitives (shadcn-generated, lightly customized)

#### `src/components/ui/button.tsx`
**Purpose:** shadcn button.
**Customization:** Add `variant="primary"` mapping to sample's `_btn1` look — solid `#377DFF`, white text, `h-12` (~48px), `rounded-[14px]`, `w-full` default in form context, slight shadow on hover.

#### `src/components/ui/input.tsx`
**Purpose:** shadcn input.
**Customization:** Raise default `h-12`, `px-4`, `text-sm`. No variant change — sample's underline-only treatment is achieved via class composition where needed.

#### `src/components/ui/label.tsx`
**Purpose:** shadcn label (Radix). Default.

#### `src/components/ui/checkbox.tsx`
**Purpose:** shadcn checkbox (Radix).
**Note:** Sample's "Remember me" and "I agree to terms" are rendered as `<input type="radio">` in HTML but function as binary toggles. We render `<Checkbox>` for semantic correctness. Visually styled to match the sample's circular indicator.

#### `src/components/ui/form.tsx`
**Purpose:** Form primitives ported for TanStack Form.
**Exports:** `<FormField form name>` (render-prop child), `<FormItem>`, `<FormLabel>`, `<FormControl>`, `<FormMessage>`, `<FormDescription>`.
**Differs from shadcn upstream:** no `<FormProvider>`/`Controller` indirection. Component takes `form` (TanStack Form instance) + `name` (field name) and renders via render-prop. `<FormMessage>` reads `field.state.meta.errors`.

### Auth Feature (`src/features/auth/`)

#### `extract-error.ts`
**Purpose:** Normalize unknown errors into `{ status, message }`.
**Exports:** `extractApiError(err: unknown): { status: number | null; message: string }`.
**Behavior:** If `err instanceof ApiError` (from `@/lib/api-error`), returns `err.status` + best `body.message || body.error || "Something went wrong."`. Otherwise `{ status: null, message: "Something went wrong." }`.

#### `use-login.ts`
**Purpose:** Mutation wrapper around generated `useLogin`.
**Exports:** `useLoginMutation(redirectParam?: string)`.
**On success:** `setAccessToken(data.data.accessToken)`, `useAuthStore.setUser(data.data.user)`, `window.location.href = safeRedirectPath(redirectParam) ?? "/"`. `isInitialized` is not touched — it was set to `true` by `initializeAuth` during the cold-boot that landed the user on `/auth/login`, so it stays `true`.
**Error surface:** returns the `mutation` object; caller reads `mutation.error` and dispatches inline-vs-toast logic.

#### `use-signup.ts`
**Purpose:** Same shape as `use-login`. Signature `useSignupMutation()` (no redirect param — signup always lands at `/`). Wraps generated `useSignup`; success returns `AuthSession` per OpenAPI, so success flow is identical to login except the post-auth destination is hardcoded `/`.

#### `login-form.tsx`
**Purpose:** The login form.
**Props:** `redirectParam?: string`.
**Fields:** `email`, `password`, `rememberMe` (decorative checkbox).
**Validation:** TanStack Form `validatorAdapter` running `loginBodySchema.safeParse` on blur and submit.
**Submit handler:**
- `mutation.mutate({ data: { email, password } })`.
- On success: hooked in `useLoginMutation`'s `onSuccess`.
- On error: `extractApiError`; if `status === 401`, set local state `inlineError = "Invalid email or password"` + `toast.error(message)` and clear password field. Else toast only.
**"Forgot password?" link:** `<button type="button">` → `toast.info("Password reset coming soon")`.
**Submit button:** primary variant, full-width, label "Login now", spinner replaces label when pending. Disabled when form invalid.

#### `signup-form.tsx`
**Purpose:** The signup form.
**Props:** none.
**Fields:** `firstName`, `lastName`, `email`, `password`, `confirmPassword`, `agreeToTerms`.
**Validation:** Compose `signupBodySchema` with two extensions:
- `.extend({ confirmPassword: z.string(), agreeToTerms: z.literal(true) })`
- `.refine((d) => d.password === d.confirmPassword, { message: "Passwords do not match", path: ["confirmPassword"] })`
**Submit handler:** `mutation.mutate({ data: { firstName, lastName, email, password } })` (drops UI-only fields).
**On error:** `extractApiError`; if `status === 409`, set inline error on `email` field via `form.setFieldMeta("email", …)` + toast. Else toast only.
**Submit button:** primary variant, full-width, label "Sign up now" (corrected from sample's "Login now" typo).

## Data Flow

### Login (happy path)

1. User visits `/auth/login?redirect=/some/path` (or none).
2. `auth.tsx` `beforeLoad` sees not authenticated → renders `<AuthLayout><Outlet /></AuthLayout>`.
3. `login.tsx` reads `redirect` from search, passes to `<LoginForm>`.
4. TanStack Form validates on blur via `loginBodySchema`; inline `<FormMessage>` per field.
5. User submits → `useLoginMutation().mutate({ data: { email, password } })`.
6. Generated `useLogin` calls `POST /api/auth/login` via auth-hooked ky.
7. Backend returns `200 { data: { accessToken, user } }`.
8. `onSuccess`:
   - `setAccessToken(data.data.accessToken)`
   - `useAuthStore.getState().setUser(data.data.user)`
   - `window.location.href = safeRedirectPath(redirectParam) ?? "/"`
9. Hard reload → `__root.tsx` `beforeLoad` → `initializeAuth()` re-hydrates from the refresh cookie now in place. No race with router guards.

### Signup (happy path)

Same shape; `POST /api/auth/signup` → `201 { data: AuthSession }`. Post-success destination is hardcoded `/` (no `redirect` param flows through signup).

### Error matrix

| Source | Trigger | Display |
|---|---|---|
| zod `safeParse` | Field-level invalid (email format, length) | Inline `<FormMessage>` below input. No toast. |
| zod cross-field (signup) | `confirmPassword !== password` | Inline under `confirmPassword`. |
| Login 400 | Backend validation | Toast `body.message`. |
| Login 401 | Invalid credentials | Inline banner above form + toast. Password field cleared, email retained. |
| Signup 400 | Validation | Toast. |
| Signup 409 | Email already in use | Inline on `email` field via `setFieldMeta` + toast. |
| 5xx | Server error | Toast (message or fallback). |
| Network / timeout | Already handled | `notifyConnectivityIssueIfNeeded` in `api-client.ts` fires once; no duplicate. |
| Google btn / Forgot password / Remember-me click | UI interaction | `toast.info("…coming soon")`. No API call. |

### Loading states

- Submit button: `<Loader2 className="animate-spin" />` from `lucide-react` replaces the label when `mutation.isPending`. Button disabled.
- Inputs remain enabled during pending — user can correct typos. Form's `onSubmit` short-circuits if `isPending`.

### Redirect safety

Uses existing `safeRedirectPath`. Blocks: non-strings, paths not starting with `/`, protocol-relative (`//…`), backslash tricks, and any path in `AUTH_ROUTES` (prevents auth-route loops).

## Error Handling

| Layer | Strategy |
|---|---|
| zod validation | Inline `<FormMessage>` only |
| `ApiError` 4xx/5xx | `extractApiError` → caller picks inline + toast vs. toast-only per error class (see matrix) |
| Network / timeout | Existing `notifyConnectivityIssueIfNeeded` (no change) |
| Disabled CTAs | `toast.info` from button click handler |
| Mutation hook crash (unexpected) | Surface as toast via `extractApiError` fallback message |

## Testing Strategy

No automated tests in Phase A — auth forms are mostly composition and thin mutation wrappers; behavioral surface is small enough that manual gate is sufficient.

**Manual smoke gate** (all must pass before declaring Phase A complete):

1. `pnpm typecheck` → 0 errors.
2. `pnpm lint` → 0 errors.
3. `pnpm dev`, then in browser at `http://localhost:3000`:
   - Hard reload at `/` while logged out → lands on `/auth/login`.
   - **Visual parity check:** open `sample_screens/login.html` in a second tab at 1440px wide. Layouts indistinguishable side-by-side (split 8/4, hero image left, form right, shape SVGs in corners, logo top of form, button proportions match).
   - Repeat for `/auth/signup` vs. `sample_screens/registration.html`.
   - At `<1024px` viewport, hero hides, form centers.
4. **Login form behavior:**
   - Submit empty → inline zod errors on email + password. No toast.
   - Invalid email format → inline after blur.
   - Valid email + short password → inline "Password must be at least 8 characters."
   - Valid credentials (against a known test account) → spinner → hard reload → lands at `/`.
   - URL `/auth/login?redirect=/foo` + valid creds → hard reload → lands at `/foo`.
   - URL `/auth/login?redirect=//evil.com` + valid creds → lands at `/` (open-redirect blocked).
   - Valid email + wrong password → inline banner above form + toast. Password cleared; email retained.
   - Click "Forgot password?" → `toast.info`. No navigation.
   - Click Google button → `toast.info`. No submit.
5. **Signup form behavior:**
   - Submit empty → inline zod errors on all required fields including terms checkbox.
   - Mismatched passwords → inline on `confirmPassword`.
   - New email → success → hard reload → `/`.
   - Existing email (run twice) → inline under email + toast.
6. **Auth-hooked client untouched:** sign in, click Logout from existing authenticated layout, lands back at `/auth/login` (existing behavior).

## Build / Execution Sequence

1. **shadcn init.** `pnpm dlx shadcn@latest init` with style `new-york`, base color `slate`, CSS variables on. Adds `class-variance-authority`, `clsx`, `tailwind-merge`, `tw-animate-css`. Creates `components.json` and `src/lib/utils.ts`.
2. **Install primitives.** `pnpm dlx shadcn@latest add button input label checkbox`. Pulls Radix peers (`@radix-ui/react-label`, `@radix-ui/react-checkbox`, `@radix-ui/react-slot`). The `form` primitive is **not** installed via shadcn (its upstream version targets react-hook-form); instead `src/components/ui/form.tsx` is hand-written in step 5 for the TanStack Form API.
3. **Customize tokens.** Append `@theme` block to `src/index.css` with the variables in §Foundation.
4. **Customize primitives.** Add `primary` variant to `ui/button.tsx`; bump default size on `ui/input.tsx`.
5. **Hand-write `ui/form.tsx`** for TanStack Form: `<FormField form name>` (render-prop child), `<FormItem>`, `<FormLabel>`, `<FormControl>`, `<FormMessage>`, `<FormDescription>`. No `<FormProvider>` indirection. `<FormMessage>` reads from `field.state.meta.errors`.
6. **Copy assets** from `sample_screens/assets/images/` to `public/auth/`: `shape1.svg`, `shape2.svg`, `shape3.svg`, `dark_shape.svg`, `dark_shape1.svg`, `dark_shape2.svg`, `login.png`, `registration.png`, `registration1.png`, `google.svg`. Copy `logo.svg` → `public/logo.svg`. Copy `logo-copy.svg` → `public/logo-dark.svg`.
7. **Build auth layout components.** `src/components/auth/`: `AuthLayout.tsx`, `AuthShapeBackdrop.tsx`, `AuthHero.tsx`, `AuthCard.tsx`, `GoogleAuthButton.tsx`, `AuthDivider.tsx`, `BottomLink.tsx`. One component per file. No domain logic.
8. **Build error helper.** `src/features/auth/extract-error.ts`.
9. **Build mutation hooks.** `src/features/auth/use-login.ts`, `use-signup.ts`. Wrap generated `useLogin` / `useSignup`.
10. **Verify generated zod schema names.** Inspect `src/gen/api/zod/`. Confirm `loginBodySchema`, `signupBodySchema` (or actual exported names) exist. Adjust import paths in step 11 if names differ.
11. **Build forms.** `src/features/auth/login-form.tsx`, `signup-form.tsx`. TanStack Form + zod + `ui/form`.
12. **Replace route bodies.** `auth.tsx` (layout swap), `auth/login.tsx`, `auth/signup.tsx` (real compositions).
13. **Verify.** `pnpm typecheck` → 0 errors. `pnpm lint` → 0 errors. `pnpm dev`, walk through smoke gate in §Testing Strategy.

## Open Items / Risks

- **TanStack Form recipe port (step 5).** Trickiest novel work in Phase A. Mitigation: minimal `<FormField>` with render-prop child + `field.state.meta.errors` read. ~30 lines, no `FormProvider`.
- **Generated zod schema names.** Step 10 verifies; import paths adjust if Kubb emits different names than assumed.
- **Poppins vs. Inter font.** Sample is Poppins; we ship Inter. If visual review flags, swap is one `@import` + `--font-sans` change. Tracked, not blocking.
- **`_btn1` exact pixel dimensions.** Eyeballed from sample CSS at step 4; iterated during step 13 visual review.
- **Sample registration form lacks `firstName`/`lastName` inputs** (backend requires them). Adding two inputs above the email field keeps the visual rhythm of the sample's input stack.
- **Sample CTA label "Login now" on registration page** is a typo in the source HTML. Spec corrects it to "Sign up now".
- **Sample `BottomLink` on registration page** also says "Don't have an account? Create New Account" (another sample bug — that link belongs on login, not registration). Spec corrects the signup-page link to "Already have an account? Sign in" pointing at `/auth/login`.
- **Multi-tab login** (tab B does not auto-redirect after tab A signs in). Known limitation; not addressed in Phase A.
